# 代码审查清单

> 本清单用于 Code Review 时对照检查，确保代码质量

---

## 一、功能正确性

### 1.1 业务逻辑
- [ ] 功能是否完整实现了需求？
- [ ] 边界条件是否都处理了？（空值、极端值、非法输入）
- [ ] 时间/日期处理是否正确？（时区、夏令时）
- [ ] 业务规则是否符合需求文档？

### 1.2 API 行为
- [ ] HTTP 方法是否正确？（GET/POST/PUT/DELETE）
- [ ] HTTP 状态码是否恰当？
- [ ] 错误响应格式是否统一？

---

## 二、代码质量

### 2.1 Rust 语言特性
- [ ] 是否正确使用 `Result` 进行错误处理？
- [ ] 是否避免了 `.unwrap()` 和 `.expect()`？
- [ ] 是否使用了合适的类型？（`Option` vs `null`）
- [ ] 异步代码是否正确使用 `await`？
- [ ] 是否避免了不必要的 `unsafe` 代码块？

### 2.2 命名规范
- [ ] 模块名使用蛇形命名法 (`snake_case`)？
- [ ] 类型名使用大驼峰命名法 (`PascalCase`)？
- [ ] 函数/变量名使用蛇形命名法？
- [ ] 命名是否清晰表达意图？

### 2.3 代码结构
- [ ] 函数是否单一职责？
- [ ] 函数长度是否适中？（建议 < 50 行）
- [ ] 是否有重复代码？（DRY 原则）
- [ ] 注释是否必要且准确？
- [ ] 公共 API 是否有文档注释？

---

## 三、安全性

### 3.1 输入验证
- [ ] 用户输入是否验证？
- [ ] SQL 查询是否使用参数化？（防止 SQL 注入）
- [ ] API 请求参数是否验证类型和范围？

### 3.2 认证授权
- [ ] 是否正确验证 API Key？
- [ ] 是否有未授权的访问路径？
- [ ] 权限检查是否正确？

### 3.3 敏感数据
- [ ] 敏感信息是否记录到日志？
- [ ] API Key 是否通过环境变量注入？

---

## 四、性能

### 4.1 数据库
- [ ] 是否使用了索引？
- [ ] 是否存在 N+1 查询问题？
- [ ] 查询是否只选择必要的字段？

### 4.2 异步处理
- [ ] 是否可以使用 `join!` 并行处理独立任务？
- [ ] 是否有阻塞调用在异步上下文中？

---

## 五、可维护性

### 5.1 配置
- [ ] 配置是否外部化？（不使用硬编码）

### 5.2 可测试性
- [ ] 是否有单元测试？
- [ ] 是否有集成测试？
- [ ] 测试是否覆盖边界情况？

### 5.3 可观测性
- [ ] 是否有适当的日志？
- [ ] 日志级别是否恰当？
- [ ] 关键操作是否有追踪？

---

## 六、错误处理

### 6.1 错误类型
- [ ] 是否使用了自定义错误类型？
- [ ] 错误是否传播到适当层级？
- [ ] 是否避免了吞掉错误？

### 6.2 用户反馈
- [ ] 错误信息是否对用户友好？
- [ ] 是否记录了内部错误详情？

---

## 七、提交前自检

在提交代码前，请快速对照：

1. 功能是否完整？
2. 是否有 `.unwrap()` 或 `.expect()` 未经处理？
3. 是否有硬编码的配置？
4. 是否有调试用的 `println!` 未移除？
5. 测试是否编写/更新？

---

*使用此清单进行 Code Review，提高代码质量*
